# RAG 评估与优化

RAG 系统不能只靠感觉调参。今天觉得答案更好，明天换一批问题可能又变差。要持续优化，就需要一套评估方法。

## 为什么要分层评估

RAG 链路很长，答案不好不一定是模型问题。

可能的问题包括：

- 文档解析错了。
- chunk 切坏了。
- 检索没召回正确资料。
- 召回了但排序太靠后。
- Prompt 没约束住模型。
- 模型生成时幻觉。

所以评估要分层：检索阶段、生成阶段、端到端阶段分别看。

例如用户问“试用期有没有年假”，最终回答错了。原因可能不是模型笨，而是检索阶段召回了“正式员工年假规则”，没有召回“试用期员工规则”。如果只看最终答案，就很难定位问题。

## 检索阶段指标

### Hit Rate

命中率表示：正确资料是否出现在 TopK 结果里。

比如 Top5 里只要有目标 chunk，就算命中。

它适合回答“有没有捞到正确资料”。

例子：标准答案依据是 `chunk_12`，系统 Top5 召回了 `[chunk_03, chunk_12, chunk_20]`，这次 Hit Rate 就算命中。

### MRR

MRR 是平均倒数排名。

正确答案排第 1，比排第 5 更好。MRR 关注正确结果出现得有多靠前。

它适合评估排序质量。

同样命中 Top5，如果正确 chunk 排第 1，用户答案通常更稳；如果排第 5，后面拼 Prompt 时可能被截掉，或者被其他噪声干扰。

### Recall 与 Precision

Recall 关注该召回的有没有召回全。

Precision 关注召回结果里有多少是真的相关。

RAG 中常见取舍是：先保证召回，再用 Rerank 和 Prompt 控制噪声。

## 生成阶段指标

### 忠实度

忠实度看答案是否忠于给定资料。

如果资料没有提到某个结论，模型却写出来了，就是忠实度问题。

### 答案相关性

答案相关性看回答是否真正回应用户问题。

有时模型虽然没有编造，但回答了一堆无关内容，这也不是好答案。

## 端到端指标

端到端评估更接近用户体验。

常见指标：

- 答案正确率。
- 兜底率。
- 用户满意度。
- 人工审核通过率。
- 问题解决率。

端到端指标最好结合真实业务问题，不要只用模型自己生成的理想问题。

## 评测集怎么构建

一个评测样本可以包含：

```json
{
  "question": "用户问题",
  "expected_answer": "期望答案",
  "golden_chunks": ["chunk_id_1", "chunk_id_2"],
  "category": "问题类型"
}
```

评测集来源可以有三种：

- 人工标注：质量最高，成本也最高。
- 用户反馈：最贴近真实场景，但需要清洗。
- 大模型辅助生成：效率高，但要人工抽查校准。

建议先做一个小而准的评测集，比如 50 到 200 条高频问题，再逐步扩展。

第一版评测集可以直接从真实客服问题里挑：退款、发票、合同、权限、常见报错各选一些。每条都标好“期望召回哪几段资料”，这样才能评估检索是不是对的。

## LLM-as-Judge

LLM-as-Judge 是让大模型当裁判，对答案进行评分。

它适合提高评估效率，但不能完全替代人工。

使用时要注意：

- 评分标准要明确。
- 裁判模型可能有偏见。
- 要抽样做人审校准。
- 不能让同一个模型既生成又无监督地评自己。

## 优化闭环

评估的价值在于定位问题。

可以按结果反推：

- Hit Rate 低：检索没召回，检查 chunk、embedding、query rewrite、TopK。
- MRR 低：召回了但排序差，考虑 Rerank 或融合策略。
- 忠实度低：检查 Prompt、引用约束、temperature、上下文噪声。
- 答案相关性低：检查意图识别、问题改写、Prompt 输出要求。
- 兜底率过高：可能检索太弱，也可能 Prompt 太保守。

## 持续评估建议

- 每次调整 chunk 策略后跑评测。
- 更换 Embedding 模型后跑评测。
- 修改 Prompt 后跑评测。
- 新增知识库数据后抽样评测。
- 记录每次评测配置，避免只记得分不记得原因。

RAG 优化不是一次性工作，而是“评估 -> 定位 -> 修改 -> 再评估”的循环。

比如你把 chunkSize 从 500 调到 1000 后，主观感觉答案更完整了，但评测发现 Hit Rate 下降、MRR 变差，说明 chunk 变大后检索粒度太粗。这个时候就不是继续调 Prompt，而是回头调整分块策略。
