# SSE 协议与流式响应

大模型聊天产品通常不是等完整答案生成后再展示，而是边生成边返回。这个体验背后常用的就是 SSE。

## SSE 是什么

SSE 是 Server-Sent Events，服务端事件推送。

它的特点是：

- 基于 HTTP。
- 服务端可以持续向客户端发送事件。
- 客户端用 `EventSource` 接收。
- 适合单向流式推送。

大模型输出天然是“一个 token 一个 token 生成”，所以很适合用 SSE 返回增量内容。

用户看到的效果是答案一点点出现；工程上看到的其实是一段段事件流。前端每收到一段 `data`，就把文字追加到聊天气泡里。

## SSE 基本格式

SSE 本质上是一些文本行。

常见字段：

```text
event: message
id: 1
retry: 3000
data: hello

```

注意最后的空行表示一个事件结束。

字段含义：

- `data`：事件数据。
- `event`：事件类型。
- `id`：事件 ID，可用于断线重连。
- `retry`：客户端重连间隔。
- 以 `:` 开头的行：注释，常用于心跳。

## SSE、WebSocket、长轮询怎么选

| 方案 | 特点 | 适合场景 |
| --- | --- | --- |
| SSE | HTTP 单向推送，简单稳定 | 大模型流式输出、通知 |
| WebSocket | 双向通信，连接长期保持 | 实时协作、游戏、双向消息 |
| 长轮询 | 客户端反复请求 | 兼容旧系统，简单通知 |

大模型 API 多选择 SSE，是因为模型主要是服务端持续输出给客户端，不需要复杂双向通信。

## 大模型流式响应

非流式响应一般是一次返回完整内容。

流式响应会返回多个 chunk，每个 chunk 里可能只有一小段增量文本。客户端需要不断解析、拼接，直到收到结束标记。

工程上不要假设每个 chunk 都有正文。有些 chunk 可能只包含角色、结束原因或 token 统计。

一个简化后的流式片段可能是：

```text
data: {"delta":"订单"}

data: {"delta":"正在"}

data: {"delta":"配送中"}

data: [DONE]
```

前端最终拼出来的是“订单正在配送中”。

## 后端为什么需要中间层

前端不建议直接连大模型 API，原因有三个：

- API Key 会暴露。
- 认证、限流、审计不好做。
- RAG 场景还需要后端先检索、组装 Prompt。

更合理的链路是：

```text
前端 -> Spring Boot 后端 -> 大模型 API
```

后端负责接入模型、转发流式内容、记录日志和处理异常。

如果是 RAG 问答，后端还可以先发送 `source` 事件，把引用资料返回给前端；再发送 `message` 事件持续输出答案。这样用户一边看到回答，一边也能看到依据来源。

## Spring Boot 中的 SseEmitter

Spring MVC 里可以用 `SseEmitter` 实现 SSE。

它可以：

- 创建一个长连接响应。
- 分多次发送事件。
- 在完成或异常时关闭连接。
- 设置超时时间。

核心不是 API 有多复杂，而是连接生命周期要处理好。

## 生产环境注意点

### 超时

流式回答可能持续几十秒，默认超时经常不够。后端、网关、Nginx 都要检查超时配置。

### 心跳

长时间没有数据时，代理或浏览器可能断开连接。可以定期发送注释行作为心跳。

### 客户端断开

用户关闭页面后，后端要能检测并停止继续转发，避免资源泄漏。

### 线程池

不要无限创建线程处理 SSE。生产环境要有明确线程池和连接数限制。

### Nginx 缓冲

反向代理可能会缓冲响应，导致前端不是实时收到。SSE 场景通常要关闭代理缓冲。

## 常见坑

- 把流式数据当成完整 JSON 解析。
- 没处理空 delta。
- 没有结束事件，前端一直等待。
- 客户端断开后后端仍继续请求模型。
- 没有记录完整回答，排查问题困难。

## 推荐事件设计

可以把服务端事件分成几类：

- `message`：正文增量。
- `source`：RAG 引用来源。
- `error`：异常信息。
- `done`：回答结束。

这样前端展示会更清晰，后端也更容易维护。

例如：

```text
event: source
data: {"title":"员工手册","page":12}

event: message
data: {"text":"根据员工手册，正式员工..."}

event: done
data: {}
```
