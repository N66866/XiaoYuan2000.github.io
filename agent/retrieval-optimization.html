# 检索策略与召回优化

# 纯向量检索的短板

向量检索擅长语义相似，但它也有明显短板：

  • 精确编号容易丢，比如合同编号、工单号、产品型号。
  • 专有名词和缩写可能识别不好。
  • 数字、日期、版本号不一定匹配准确。
  • 用户明确输入关键词时，向量检索可能召回语义相近但关键词不匹配的内容。

所以检索优化的方向不是“只用向量”，而是把不同检索方式组合起来。

比如用户问“ERR-10023 是什么错误”。向量检索可能觉得“系统错误码说明”相关，但不一定精准命中 ERR-10023；BM25 看到这个精确编号，反而能直接把包含该错误码的文档捞出来。

# BM25 是什么

BM25 是经典关键词检索算法。可以理解为更聪明的关键词匹配。

它主要考虑三件事：

  • 词频：关键词出现越多，相关性越高，但不会无限增加。
  • 逆文档频率：越稀有的词，区分度越高。
  • 文档长度归一化：长文档不能因为词多就天然占便宜。

BM25 适合找精确关键词、编号、术语、缩写。

例如“QPS 限流配置”“合同编号 HT-2026-001”“发票代码 044031900111”这类问题，关键词本身就很关键，BM25 往往比纯向量更稳。

# 向量检索与 BM25 的关系

两者不是替代关系，而是互补关系。

检索方式 擅长 不擅长
向量检索 语义相似、同义表达、自然语言问题 编号、精确词、数字、版本
BM25 精确关键词、术语、编号 同义词、语义理解、上下文

一个成熟 RAG 系统一般会做混合检索。

一个常见组合是：用户问“买错了怎么退”，向量检索能召回“退货规则”；用户问“SKU-A1024 支持退吗”，BM25 能确保命中 SKU-A1024 这个型号。混合后，两类问题都更稳。

# 混合检索

混合检索就是同时使用向量检索和关键词检索，然后合并结果。

常见方案：

  • Milvus 原生混合检索：系统更简单，适合希望减少组件数量的场景。
  • Elasticsearch + Milvus：ES 做关键词检索，Milvus 做向量检索，适合已有 ES 基础或搜索需求很复杂的场景。

如果项目刚起步，优先选择组件更少的方案。只有当关键词检索、复杂过滤、搜索运营能力成为重点时，再考虑双系统。

# 分数融合的问题

向量检索返回的是相似度分数，BM25 返回的是关键词相关性分数。两种分数不在同一个尺度上，不能简单相加。

例如：

  • 向量相似度可能是 0.78。
  • BM25 分数可能是 12.5。

直接加权很容易失真。

# RRF：倒数排名融合

RRF 是 Reciprocal Rank Fusion，倒数排名融合。

它不关心原始分数，只关心排名。一个结果在多个检索列表里排名都靠前，就会得到更高融合分。

它的好处是：

  • 简单。
  • 鲁棒。
  • 不要求两种检索分数可比。
  • 很适合混合检索的第一版实现。

工程上可以先分别取向量 TopK 和 BM25 TopK，再用 RRF 合并，最后进入重排序。

例如向量检索把文档 A 排第 1、文档 B 排第 3；BM25 把文档 B 排第 1、文档 A 排第 5。RRF 会认为 A 和 B 都值得保留，因为它们都在某一路检索中靠前。

# Rerank 重排序

检索阶段通常追求“先多召回一点”，但 TopK 里可能有噪声。Rerank 的作用是对候选结果做更精细的排序。

可以这样理解：

  • Embedding 检索是快速粗筛。
  • Rerank 是慢一点但更准的精排。

常见模型结构：

  • Bi-Encoder：问题和文档分别向量化，速度快，适合召回。
  • Cross-Encoder：问题和文档一起输入模型，判断相关性更准，但速度慢，适合重排序。

不要直接用 Cross-Encoder 做全量检索，因为成本太高。正确做法是先召回几十条候选，再 Rerank 到最终几条。

可以把它理解成招聘筛简历：Embedding 像先从几万份简历里粗筛 100 份，Rerank 像面试官认真读这 100 份，再选出最匹配的 5 份。不能让面试官从几万份开始逐份精读。

# 推荐检索链路

一个比较稳的 RAG 检索链路：

  1. 对用户问题做改写或补全，必要时提取关键词。
  2. 并行做向量检索和 BM25 检索。
  3. 用元数据过滤权限、时间、文档类型。
  4. 用 RRF 合并候选结果。
  5. 使用 Rerank 模型精排。
  6. 取最终 TopN 拼进 Prompt。
  7. 要求模型基于资料回答，并输出来源。

# 调优优先级

RAG 效果不好时，不要只盯着模型。建议按这个顺序排查：

  1. 文档解析是否干净。
  2. Chunk 是否切得合理。
  3. 元数据过滤是否正确。
  4. Embedding 模型是否适合当前语言和领域。
  5. TopK 是否太小或太大。
  6. 是否需要 BM25 补充精确匹配。
  7. 是否需要 Rerank。
  8. Prompt 是否限制了模型依据资料回答。
