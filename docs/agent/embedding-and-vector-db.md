# Embedding 与向量数据库

## 为什么关键词检索不够

关键词检索适合精确匹配，但不擅长理解语义。

比如用户问“东西买错了怎么处理”，知识库里写的是“用户可申请退货”。这两个句子关键词不完全一样，但语义相关。

关键词检索常见问题：

- 同义词：退货、退款、不要了、买错了。
- 一词多义：苹果可能是水果，也可能是公司。
- 上下文缺失：只看词频，不理解整句话的意思。

所以 RAG 需要语义检索，而 Embedding 就是语义检索的基础。

## Embedding 是什么

Embedding 是把文本转换成向量的过程。

向量可以理解为一组数字坐标。语义相近的文本，在向量空间里距离更近；语义无关的文本，距离更远。

例如：

- “如何申请退款”
- “买错商品可以退吗”

这两句话在关键词上不完全一样，但 Embedding 后的向量应该比较接近。

反过来，“苹果手机怎么退款”和“苹果有哪些营养价值”都包含“苹果”，关键词检索可能觉得相关，但 Embedding 会结合上下文，把一个理解成电子产品售后，一个理解成水果知识。

## Embedding 模型怎么选

选择 Embedding 模型时，重点看：

- 中文效果：中文知识库优先选中文或多语言表现好的模型。
- 向量维度：维度越高不一定越好，要结合成本和检索效果。
- 最大输入长度：能不能处理较长 chunk。
- 性能和成本：入库量大时，Embedding 成本很明显。
- 是否支持本地部署：私有化场景可能需要。

常见选择包括商业 API 模型和开源模型。中文场景可以重点关注 Qwen Embedding、bge、m3e 等方向。

## 相似度计算

最常用的是余弦相似度。

它关注两个向量的方向是否接近，而不是绝对长度。分数越高，表示语义越相似。

实际使用时，不要迷信固定阈值。相似度分数会受这些因素影响：

- Embedding 模型
- chunk 长度
- 问题表达方式
- 领域术语
- 是否做了文本清洗

阈值需要结合业务数据测试。

例如你在测试集中发现：相似度 0.82 以上基本都相关，0.65 到 0.75 之间有一半是噪声，那就可以先把召回阈值放在 0.75 左右，再通过 Rerank 或人工评测继续调整。不要看到网上说“0.8 是好阈值”就直接照搬。

## 为什么需要向量数据库

最简单的想法是把向量存到 MySQL，然后每次查询时逐条计算相似度。

小数据量可以这么玩，但数据一大就会很慢。比如几十万、几百万个 chunk，每次都全量比较，性能不可接受。

向量数据库的核心价值是：用索引结构加速相似向量检索。

可以类比外卖骑手找地址：暴力搜索像从城市第一栋楼开始挨个问；向量索引像先判断大概在哪个区，再进入小区找楼栋，速度会快很多，但也可能因为分区判断不准漏掉少数结果。

## ANN：近似最近邻搜索

ANN 是 Approximate Nearest Neighbor，近似最近邻搜索。

它不保证每次都找到绝对最相似的结果，但能用很高速度找到足够相似的候选集。

RAG 场景通常接受这种取舍：检索速度和整体召回效果，比理论上的绝对精确更重要。

## 常见索引思路

### IVF

IVF 可以理解为先把向量分区，再到最可能相关的分区里查。

优点是简单、速度快。缺点是如果分区不准，可能漏掉真正相关的结果。

### HNSW

HNSW 是现在很常见的向量索引算法。它可以理解为一张多层导航图：

- 上层图节点少，用来快速跳到大概区域。
- 下层图节点多，用来精细查找。

它的检索速度很快，召回效果也好。代价是内存占用较高，构建索引也需要成本。

## 向量数据库怎么选

常见方案分三类：

- 专用向量数据库：Milvus、Qdrant、Weaviate 等。
- 搜索引擎扩展：Elasticsearch / OpenSearch 的向量能力。
- 传统数据库扩展：PostgreSQL + pgvector 等。

选型可以按这个思路：

- 数据量小、系统简单：pgvector 可能够用。
- 已经重度使用 ES：可以先评估 ES 向量检索。
- 数据量大、检索是核心能力：选择专用向量数据库。
- 需要混合检索和扩展能力：关注 Milvus、ES 等生态支持。

## Milvus 核心概念

Milvus 可以类比传统数据库理解：

- Collection：类似表。
- Schema：类似表结构。
- Vector Field：存向量的字段。
- Scalar Field：存普通字段，比如文件名、部门、时间。
- Index：向量索引。
- Partition：分区，可以按业务、租户、时间等拆分。

RAG 中通常会把 chunk 文本、向量、元数据一起写入 Collection。检索时既查相似度，也用元数据做过滤。

一个 chunk 在向量库里大概长这样：

```json
{
  "chunk_id": "doc-001-0003",
  "content": "正式员工入职满一年后，每年享有 5 天年假。",
  "vector": [0.12, -0.08, 0.31],
  "file_name": "员工手册.pdf",
  "department": "all",
  "version": "2026"
}
```
