# Java Agent：RAG 知识库管理宏观设计

这一篇开始进入 Java Agent 开发篇。前面的扫盲篇讲的是 RAG、Embedding、Chunk、Metadata 这些概念；真正落到项目里，首先要回答的是：知识库到底要怎么建模，文档怎么处理，分块和向量怎么关联，失败后怎么恢复。

## 从理论到实战的鸿沟

理论上的 RAG 流程很简单：

```text
文档 -> 解析 -> 分块 -> 向量化 -> 入库 -> 检索 -> 回答
```

但工程里会多出一堆问题：

- 一个用户可以有多个知识库吗？
- 文档上传后是立刻分块，还是用户手动触发？
- URL 文档怎么定时刷新？
- 分块失败后文档状态怎么回滚？
- 关系型数据库和向量库如何保持一致？
- 文档禁用后，向量库里的数据要不要删除？
- 分块策略能不能按文档单独配置？

所以项目里不能只有一个 `chunk` 表，至少要把知识库、文档、分块三层模型拆出来。

## 三层架构

### 用户层

用户层负责前端交互：

- 创建知识库。
- 上传文档。
- 查看文档处理状态。
- 手动触发分块。
- 启用或禁用文档。
- 管理 chunk。
- 配置 URL 文档定时同步。

这一层重点是让用户知道“文档现在处于什么状态”，否则上传后后台异步跑任务，用户会完全不知道发生了什么。

### 业务层

业务层负责核心流程：

- 参数校验。
- 文档状态流转。
- 文件上传到对象存储。
- 发送 MQ 消息。
- 执行文档解析、分块、向量化。
- 写入数据库和向量库。
- 处理失败、重试和日志。

这一层是 RAG 知识库系统的核心。

### 存储层

存储层一般分三类：

- MySQL：存知识库、文档、chunk 元数据、任务状态。
- 对象存储：存原始文件。
- 向量数据库：存向量和检索字段。

这里有一个重要设计原则：关系型数据库是业务事实来源，向量库是检索索引。不要把所有业务状态都塞进向量库。

## 核心概念模型

### 知识库

知识库是文档集合的容器。

常见字段：

- `id`
- `name`
- `description`
- `owner_id`
- `embedding_model`
- `chunk_config`
- `enabled`
- `created_at`
- `updated_at`

为什么需要知识库概念？

因为不同知识库可能有不同用途：

- 公司制度知识库。
- 产品手册知识库。
- 技术文档知识库。
- 客服 FAQ 知识库。

它们可能使用不同的 Embedding 模型、分块参数、权限范围和更新策略。

### 知识库文档

文档是知识库里的一个文件或一个远程资源。

常见字段：

- `id`
- `knowledge_base_id`
- `name`
- `source_type`
- `source_url`
- `object_key`
- `file_hash`
- `process_mode`
- `status`
- `enabled`
- `chunk_config`
- `schedule_config`

文档要同时支持两类来源：

- `FILE`：用户上传本地文件。
- `URL`：从远程链接抓取文件。

处理模式也可以拆成两类：

- `CHUNK`：直接按分块策略处理。
- `PIPELINE`：进入更复杂的处理流水线，比如 OCR、表格抽取、清洗、审核。

### 文档状态

文档状态是非常关键的工程字段。

可以设计成：

```text
UPLOADED -> CHUNKING -> CHUNKED -> FAILED
```

也可以再细一点：

```text
INIT
UPLOADED
PARSING
CHUNKING
EMBEDDING
INDEXING
READY
FAILED
```

状态越细，排查越方便；状态越粗，实现越简单。实际项目可以先用粗粒度状态，再用分块日志记录细节。

### 知识库分块

Chunk 是文档处理后的最小检索单元。

常见字段：

- `id`
- `document_id`
- `knowledge_base_id`
- `chunk_index`
- `content`
- `content_hash`
- `token_count`
- `metadata`
- `enabled`
- `created_at`
- `updated_at`

向量字段不一定要放在 MySQL。更推荐：

- MySQL 存 chunk 文本、状态、元数据。
- 向量库存 vector 和检索字段。

这样业务管理和向量检索职责更清楚。

## 表设计建议

### 知识库表

`t_knowledge_base` 负责知识库基本信息。

设计要点：

- 保存默认分块配置。
- 保存默认 Embedding 模型。
- 支持启用、禁用。
- 支持逻辑删除。

### 文档表

`t_knowledge_document` 负责文档生命周期。

设计要点：

- `source_type` 区分本地文件和 URL。
- `object_key` 指向对象存储文件。
- `status` 表示处理状态。
- `process_mode` 表示处理方式。
- `chunk_config` 允许文档覆盖知识库默认分块配置。
- `schedule_config` 支持 URL 文档定时刷新。

### 分块表

`t_knowledge_chunk` 负责 chunk 管理。

设计要点：

- `chunk_index` 保证展示顺序。
- `content_hash` 用于判断内容是否变化。
- `enabled` 控制是否参与检索。
- `metadata` 保存页码、标题层级、来源等扩展信息。

## 文档处理流程

完整链路可以拆成：

```text
上传文件
  -> 保存对象存储
  -> 创建文档记录
  -> 用户触发分块
  -> CAS 更新文档状态
  -> 发送 MQ 消息
  -> Consumer 解析文档
  -> 分块
  -> 调 Embedding
  -> 写 MySQL chunk
  -> 写向量库
  -> 更新文档状态
```

上传和分块拆成两个动作是有价值的：

- 上传只负责拿到原始文件。
- 分块是耗时任务，适合异步执行。
- 用户可以先上传多个文档，再批量处理。
- 上传成功但分块失败时，文档仍然可以保留。

## 异常处理

常见失败点：

- 文件上传对象存储失败。
- 数据库插入失败。
- MQ 消息发送失败。
- 文档解析失败。
- Embedding API 超时。
- 向量库写入失败。

处理建议：

- 上传阶段失败：直接返回错误，不创建文档或清理已上传文件。
- 分块阶段失败：记录失败日志，文档状态改为 `FAILED`。
- 部分写入失败：使用先删后插或补偿任务保证一致性。
- 可重试失败：进入重试队列。
- 不可重试失败：暴露错误原因给管理端。

## 设计重点

知识库管理不是简单 CRUD，它是一个异步数据处理系统。核心是三件事：

- 状态清晰：用户和系统都知道文档现在在哪一步。
- 边界清晰：上传、分块、向量化、入库职责分开。
- 一致性清晰：MySQL、对象存储、向量库之间要有补偿策略。
