# Java Agent：知识库定时同步与故障恢复

URL 类型文档和外部知识源会变化。如果知识库只支持手动上传，内容很快会过期。定时同步要解决的是：按计划检测远程文档变化，必要时重新抓取、分块、向量化，并且在多实例部署下不重复执行。

# 为什么需要定时同步

适合定时同步的场景：

  • 远程 PDF 手册。
  • Git raw 文档。
  • 企业内部文档 URL。
  • 产品说明页面。
  • 公开政策文件。

用户不希望每次远程文件变化后都手动上传。系统应该能按 cron 表达式自动刷新。

# 表设计

# 任务主表

t_knowledge_document_schedule 保存调度配置。

关键字段：

  • id
  • document_id
  • cron
  • enabled
  • next_fire_time
  • lock_owner
  • lock_until
  • last_exec_status
  • created_at
  • updated_at

它描述“任务什么时候该跑”。

# 执行历史表

t_knowledge_document_schedule_exec 保存每一次执行记录。

关键字段：

  • id
  • schedule_id
  • document_id
  • status
  • phase
  • started_at
  • finished_at
  • error_message
  • old_file_hash
  • new_file_hash

它描述“这次跑得怎么样”。

为什么要历史表？

  • 排查失败。
  • 统计同步耗时。
  • 判断是否长期失败。
  • 保存每次变更结果。

# 核心组件

# 调度入口

KnowledgeDocumentScheduleJob 周期性扫描到期任务。

职责：

  • 查询 next_fire_time <= now 的任务。
  • 尝试获取锁。
  • 提交线程池执行。

# 锁管理器

ScheduleLockManager 负责分布式锁。

这里可以使用数据库租约锁：

UPDATE schedule
SET lock_owner = ?, lock_until = ?
WHERE id = ?
  AND (lock_until IS NULL OR lock_until < NOW())

# 刷新处理器

ScheduleRefreshProcessor 负责真正执行刷新：

  • 校验任务。
  • 检测远程文件变化。
  • 下载新文件。
  • 更新对象存储。
  • 执行分块。
  • 更新文档元数据。

# 状态管理器

ScheduleStateManager 负责状态流转和执行记录。

不要把状态更新散落在各处，否则失败后很难知道卡在哪。

# 为什么用数据库锁

定时同步任务本身已经围绕 MySQL 状态表运转。用数据库租约锁的好处：

  • 不额外依赖 Redis。
  • 锁状态可观察。
  • 和任务记录在同一个存储里。
  • 宕机后可以靠 lock_until 自动过期。

缺点是吞吐不如 Redis 锁，但定时同步通常不是高频场景，可以接受。

# 租约锁与心跳

租约锁不是永久锁，而是带过期时间。

执行任务时需要心跳续期：

抢锁成功 lock_until = now + 60s
任务执行中每 20s 续期一次
任务完成后释放锁

# 远程文件变更检测

不要每次都完整下载远程文件。可以做两级检测。

# 第一级：HEAD 探测

通过 HEAD 请求读取：

  • ETag
  • Last-Modified
  • Content-Length
  • Content-Type

如果这些信息没有变化，可以跳过下载。

# 第二级：下载后 hash

有些服务器不返回 ETag 或 Last-Modified，或者返回不可靠。此时需要下载文件，计算内容 hash，再和旧文件 hash 比较。

HEAD 能判断没变 -> 直接跳过
HEAD 无法判断 -> 下载临时文件 -> 计算 hash -> 决定是否刷新

# 调度执行流程

扫描到期任务
  -> 获取租约锁
  -> 提交线程池
  -> 启动心跳
  -> 校验任务和文档状态
  -> 创建执行记录
  -> 远程变更检测
  -> 无变化：更新 next_fire_time，结束
  -> 有变化：下载新文件
  -> 抢占文档运行权
  -> 执行分块和向量化
  -> 应用新文件元数据
  -> 标记成功
  -> 释放锁和清理临时文件

# 双层防并发

定时同步要防两类并发：

  • 同一个调度任务被多个实例执行。
  • 定时同步和用户手动分块冲突。

第一层用 schedule 租约锁。

第二层用文档状态 CAS，例如只有文档不在 RUNNING 状态时才能刷新。

这样即使多个入口触发，也不会同时改同一份文档。

# RUNNING 状态恢复

如果实例宕机，文档可能停留在 RUNNING 或 CHUNKING。

需要恢复机制：

扫描长时间 RUNNING 的任务
  -> 判断是否超过阈值
  -> 检查锁是否过期
  -> 标记为 FAILED 或可重试

阈值要比正常最大执行时间长，避免误杀慢任务。

# Cron 校验

调度系统要限制 cron 表达式：

  • 不能小于最小间隔。
  • 不能产生过高频率任务。
  • 保存时计算下一次执行时间。

例如不允许用户配置每 1 秒同步一次远程 PDF，否则系统会被拖垮。

# 故障处理

常见故障：

  • 远程文件不可访问。
  • 下载超时。
  • 文件 hash 计算失败。
  • 分块失败。
  • 向量库写入失败。
  • 心跳续锁失败。
  • 元数据切换失败。

处理原则：

  • 执行记录必须落库。
  • 临时文件必须清理。
  • 锁失效后不要继续写关键状态。
  • 分块失败不应该删除旧可用数据。
  • 新文件元数据切换失败时，要保留足够信息用于补偿。

# 设计重点

定时同步的核心不是 cron，而是状态和恢复。

真正要守住的是：

  • 同一任务不会多实例重复跑。
  • 同一文档不会被手动和定时同时处理。
  • 实例宕机后任务能恢复。
  • 外部文件没变时不做无意义重建。
  • 文件变更后能完整刷新索引。
