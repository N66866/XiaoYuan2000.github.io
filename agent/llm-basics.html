# 大模型基础扫盲

# 大模型是什么

大模型可以先理解成一个“会根据上下文续写的概率模型”。它不是传统程序里一堆固定的 if-else，而是通过大量文本训练，学会在不同语境下预测下一个 token 应该是什么。

传统规则系统的问题是：用户表达非常灵活。比如“我要退货”“东西不想要了”“买错了能退吗”，本质都是同一个意图，但规则写法很难穷举。大模型的优势就在于，它能从语义上理解这些表达的相似性。

举个更贴近日常开发的例子：以前做客服机器人，可能要写很多规则：

if (question.contains("退货") || question.contains("退款")) {
    return "请进入订单详情申请售后";
}

但用户可能会说“买错了咋办”“不想要了能不能寄回去”“这个尺寸不合适”。这些话没有明确出现“退货”，规则就容易漏掉。大模型的价值是，它能判断这些表达背后的意图都接近“售后退货”。

# 大模型“大”在哪里

“大”主要体现在三件事：

  • 参数量大：模型内部有大量权重参数，用来记录训练中学到的语言和知识模式。
  • 训练数据大：训练语料覆盖网页、书籍、代码、问答、论文等大量文本。
  • 计算量大：训练和推理都需要较强算力，尤其是长上下文、多轮对话和复杂推理。

参数可以粗略理解为模型内部的“经验值”。训练时，模型不断调整这些数字，让自己的预测更接近真实答案。

# 几个核心概念

# Token

Token 是大模型处理文本的基本单位。它不完全等于一个字，也不完全等于一个词。中文里常见情况是一个汉字、一个词片段或一个标点都可能变成 token。

可以粗略理解为：模型不是直接按“人眼看到的一句话”处理文本，而是先把文本拆成一小段一小段的积木。比如“我要申请退款”可能会被拆成若干 token，英文单词、中文词片段、标点都会占位置。

开发时要关心 token，因为它影响：

  • 调用成本
  • 上下文长度
  • 输出长度
  • 响应速度

# 上下文窗口

上下文窗口就是模型一次能“看见”的文本长度。你发给模型的系统提示词、历史消息、检索资料、用户问题，都要占用上下文。

RAG 系统尤其要注意上下文窗口，因为检索回来的资料不能无限塞进去。资料太少，答案不准；资料太多，成本高，甚至会把重点冲淡。

比如模型上下文只能装 8K token，你却把 20 页制度全文都塞进去，模型不仅装不下，还可能在一堆无关段落里忽略真正关键的退款规则。RAG 的检索和分块，就是为了只把最相关的几段资料送进去。

# Temperature

Temperature 控制回答的随机性。

  • 越低：越稳定，适合客服、知识库问答、代码解释。
  • 越高：越发散，适合创意写作、头脑风暴、文案生成。

做企业知识库、Agent 工具调用时，一般不要把 temperature 调太高。

一个简单类比：同样问“报销发票丢了怎么办”，企业知识库希望每次都稳定回答制度要求，这时 temperature 应该低；如果让模型帮你想 10 个活动标题，就可以适当调高，让它更发散。

# MoE

MoE 是 Mixture of Experts，混合专家架构。可以理解为模型内部有多个“专家模块”，每次请求只激活其中一部分。

它的好处是：总参数可以很大，但每次推理不必把所有参数都跑一遍，从而兼顾能力和成本。

# Base 模型与 Chat 模型

Base 模型是基础续写模型，更像“文本补全器”。Chat 模型是在 Base 模型上做了指令微调和对齐，更适合对话、问答、任务执行。

我们做 RAG 或 Agent，大多数时候调用的是 Chat 模型，因为它更懂：

  • system / user / assistant 角色
  • 按指令输出
  • 多轮对话
  • 结构化回答
  • 工具调用或函数调用

# 深度思考模式

一些模型提供“深度思考”能力，适合复杂推理、数学、代码分析、方案权衡等任务。

但它不是所有场景都需要。知识库问答、简单摘要、分类提取，通常用普通模式就够了。深度思考会带来更高延迟和更多 token 消耗。

可以按这个原则选：

  • 问事实、查资料：普通模式。
  • 做推理、排错、设计方案：考虑深度思考。
  • 做 RAG 答案生成：先保证检索资料正确，再考虑模型是否需要深度推理。

# 为什么开发者需要 API

网页端适合人直接聊天，API 适合系统集成。

有了 API，就可以把大模型接入 Java 后端、知识库、客服系统、数据分析平台、审批流等业务系统。对开发者来说，大模型不再只是聊天页面，而是系统里的一个能力组件。
