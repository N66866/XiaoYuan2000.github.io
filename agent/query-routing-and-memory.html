# 查询重写、意图路由与多轮记忆

RAG 系统做到后面，会遇到三个很现实的问题：

  • 用户的问题很口语，直接检索效果差。
  • 不是所有问题都应该走知识库。
  • 多轮对话里，模型看似有记忆，其实每次请求都是重新输入上下文。

这一篇把查询重写、意图识别和会话记忆放在一起看，因为它们都发生在“真正调用检索和生成之前”。

# 查询重写

查询重写是把用户原始问题改写成更适合检索的问题。

例如：

用户：这个怎么退？
改写：用户购买商品后如何申请退货？
1
2

它解决的是检索系统的“失忆”和“不懂口语”问题。

多轮对话里尤其明显：

用户：iPhone 15 可以退吗？
助手：可以，满足七天无理由条件即可。
用户：那运费谁出？
改写：iPhone 15 退货时，退货运费由谁承担？
1
2
3
4

如果不改写，检索系统只看到“那运费谁出”，很难知道用户还在问 iPhone 15 的退货问题。

# 查询重写的常见策略

# 指代消解

把“它”“这个”“刚才那个”还原成明确对象。

用户：它支持退款吗？
改写：某某商品是否支持退款？
1
2

# 上下文补全

结合历史对话补全省略信息。

用户：多久能到账？
改写：用户申请退款后，退款多久能到账？
1
2

# 口语转正式

把口语表达转成知识库更可能使用的术语。

买错了怎么办 -> 如何申请退货或退款
1

# 多意图拆分

一个问题里可能包含多个意图。

怎么退款，运费谁出？
1

可以拆成：

  • 如何申请退款。
  • 退货运费由谁承担。

# 关键词扩展

给查询补充同义词和业务术语，提升召回。

例如退款问题可以扩展：退货、售后、退回、订单取消。

# 查询重写的注意点

查询重写不是越多越好。它会增加一次模型调用成本，也可能把用户原意改坏。

建议：

  • 短问题、省略问题、多轮追问优先改写。
  • 已经很清晰的关键词问题可以不改写。
  • 保留原始问题和改写问题，方便排查。
  • 改写失败时回退到原始问题。
  • 对高频问题做缓存。

# 意图识别

不是所有问题都应该进入 RAG。

常见意图可以分四类：

  • 知识检索：查询制度、文档、说明。
  • 工具调用：查订单、创建工单、发送邮件。
  • 闲聊对话：打招呼、普通聊天。
  • 引导澄清：用户问题太模糊，需要追问。

如果一根筋地所有问题都走知识库，系统会变得很笨：闲聊查不到资料，工具任务只会回答文字，模糊问题会召回一堆噪声。

例如“帮我查一下订单 10086 到哪了”不应该走知识库，因为知识库没有实时订单状态；它应该路由到订单查询工具。相反，“退货规则是什么”才应该走 RAG。

# 意图识别方案

# 规则匹配

适合明确、稳定、高频的场景。

优点是快、便宜、可控。缺点是覆盖有限。

# 大模型分类

用模型判断用户意图，适合表达灵活的场景。

需要明确分类标签和输出格式，例如：

只返回以下意图之一：
knowledge_retrieval | tool_invocation | chitchat | clarification
1
2

# 混合方案

实际项目更推荐混合：

  1. 规则先处理高置信问题。
  2. 规则不确定时交给模型分类。
  3. 分类低置信时走澄清。

# 路由器模式

意图识别之后，需要一个 Router 决定下一步走哪里。

用户问题
  -> 意图识别
  -> Query 改写
  -> 路由到 RAG / 工具调用 / 闲聊 / 澄清
  -> 生成最终回复

Router 是 Agent 系统的雏形：它不只是回答问题，而是在决定“该调用什么能力”。

# 多轮对话记忆

大模型本身不会记住上一次请求。所谓记忆，本质上是每次请求时把历史消息重新放进 messages。

这会带来 token 膨胀问题：历史越多，成本越高，响应越慢，还会挤占 RAG 资料的上下文空间。

# 会话记忆策略

# 完整历史

把全部历史都带上。

适合短会话，不适合生产长期对话。

# 滑动窗口

只保留最近 N 轮。

简单实用，适合大多数聊天场景。

# Token 截断

按 token 预算截断历史。

比固定轮数更精细，但实现复杂一点。

# 摘要压缩

把旧对话总结成一段摘要，再加上最近几轮消息。

适合长会话。

# 混合策略

推荐做法是“摘要 + 最近 N 轮”：

  • 摘要保存长期背景。
  • 最近 N 轮保留对话细节。

# RAG 与记忆的上下文预算

一次 RAG 请求里，上下文通常要装三类东西：

  • 系统 Prompt
  • 会话记忆
  • 检索资料

优先级建议：

  1. 系统 Prompt 不能丢。
  2. 检索资料要保证质量。
  3. 记忆按预算裁剪。

不要为了保留聊天历史，把真正回答问题的知识库资料挤掉。

比如上下文窗口只能放 20 段内容，与其保留用户半小时前的闲聊，不如保留最近 3 轮关键对话和当前问题召回的制度条款。记忆要服务当前任务，而不是机械保存全部历史。

# 存储方案

  • 内存：适合 demo 和单机测试。
  • Redis：适合在线会话，读写快，方便设置过期时间。
  • MySQL：适合长期归档、审计和分析。

生产环境通常是 Redis 保存热会话，数据库保存必要归档。
