# LLM 概述

## 本章导读

LLM 是 Prompt Engineering、RAG、Function Calling、MCP、Agent、Workflow、Context Engineering 的共同基础。学习 LLM 概述，重点不是背模型名，而是理解三件事：

- 大模型如何工作。
- 大模型擅长什么、不擅长什么。
- 工程上如何用工具、知识库、提示词和代码补齐它的短板。

一句话总结：**大模型是一个强大的语言和推理“大脑”，但它不是数据库、计算器、实时搜索引擎，也不是能直接执行业务动作的后端系统。**

## LLM 是什么

LLM，全称 Large Language Model，大语言模型。它可以理解成一个基于上下文预测下一个 token 的概率模型。

它擅长：

- 理解自然语言。
- 生成文本。
- 总结、改写、翻译。
- 代码生成、代码解释、代码调试。
- 从非结构化文本中抽取结构化信息。
- 根据上下文完成问答、分析和简单推理。

但它不是绝对可靠的事实源。它生成的内容看起来很自然，不代表一定真实、实时、可执行或经过业务校验。

## 大模型的工作原理

### 一句话解释

大语言模型最核心的工作方式是：**根据已有上下文，预测下一个最可能出现的 token**。

比如输入“这个项目的核心亮点是”，模型会根据前文语义，计算后面出现“性能”“稳定性”“扩展性”“成本”等 token 的概率，然后选择一个 token 输出。生成出的 token 又会被追加回上下文，继续预测下一个 token。这个过程不断循环，就形成了完整回答。

所以，大模型看起来像是在写文章、聊天、翻译、写代码，底层其实都是在做连续的 next token prediction。

面试时可以这样回答：

> 大语言模型的核心是基于上下文预测下一个 token。训练阶段，它通过海量文本学习语言中的统计规律、语义关系和上下文模式；推理阶段，用户输入会变成上下文，模型不断预测下一个 token，再把生成出的 token 加回上下文继续预测。它看起来像是在理解，是因为 Transformer 的自注意力机制能让每个 token 和上下文中的其他 token 建立关系，从而捕捉语义、指代、语法和知识模式。但本质上它仍然是概率生成模型，不是传统意义上真正知道事实的数据库。

### 训练阶段：从海量文本中学习模式

训练时，模型会阅读海量文本，例如书籍、网页、论文、代码、问答和对话数据。它不是把这些内容一字不漏地背下来，而是通过反复预测和纠错，学习词语、句子、语义和上下文之间的统计规律。

对 GPT 这类生成式模型来说，更准确的训练方式是：**根据前文预测下一个 token**。例如给出“人工智能正在”，模型要预测后面可能是“改变”“影响”“推动”等 token。训练系统会拿模型预测和真实答案比较，如果预测不准，就通过反向传播调整模型参数。

课程中常见的 `[MASK]` 遮住词再预测，更接近 BERT 这类 Masked Language Model 的训练方式。这里要区分：

- BERT：偏“完形填空式理解”，适合理解、分类、匹配、抽取等任务。
- GPT：偏“从左到右续写式生成”，适合对话、写作、代码生成、工具调用等任务。

### 推理阶段：循环预测下一个 token

模型训练完成后，用户输入的 Prompt 会作为初始上下文。模型先预测第一个 token，再把这个 token 加回上下文，然后继续预测第二个 token。如此循环，直到生成结束符、达到长度上限，或者满足停止条件。

例如用户输入“写一首关于春天的诗”，模型可能先生成“春天”，上下文变成“写一首关于春天的诗 春天”；接着预测“的”，再预测“风”“吹”“过”等 token，最后组合成完整文本。

这里要注意：模型不是永远选择概率最高的 token。永远选择最高概率会让输出更稳定，但也容易死板、重复、缺少创造性。因此生成时通常会配合 `temperature` 等参数控制随机性。

### Temperature：控制输出随机性

`temperature` 越低，模型越倾向选择高概率 token，回答越稳定、保守、可复现；`temperature` 越高，模型越容易选择低概率但有创造性的 token，回答更发散，也更可能跑偏。

常见设置可以这样记：

- 知识库问答：`0` 到 `0.2`，追求稳定、准确、忠实资料。
- 代码生成：`0` 到 `0.3`，越接近生产代码越应该保守。
- 创意写作：`0.7` 到 `1.0`，需要更多变化和想象力。

在 RAG、工具调用、客服问答这类工程场景里，一般不要把 temperature 调太高，否则会增加幻觉和格式不稳定风险。

### 为什么它看起来像理解了上下文

“预测下一个 token”听起来很简单，但模型不是只看前一个词，而是会综合整个上下文。

例如：

- “苹果很好吃”里的“苹果”更可能指水果。
- “苹果发布了新手机”里的“苹果”更可能指公司。

模型能区分这两种含义，关键来自 Transformer 的自注意力机制。自注意力会让每个 token 都去计算它和上下文中其他 token 的相关性权重。处理“苹果”时，模型会发现它和“好吃”关系更强，于是语义偏水果；如果它和“发布”“手机”关系更强，语义就偏公司。

多头注意力可以理解为从多个角度同时看关系：有的头关注主谓宾关系，有的头关注指代关系，有的头关注修饰关系，有的头关注长距离依赖。多个注意力头合在一起，模型就能更全面地理解上下文。

## Transformer

### 为什么需要 Transformer

Transformer 来自 2017 年 Google 的论文《Attention Is All You Need》，它是现代大语言模型的基础架构。GPT 中的 `T` 就是 Transformer。

在 Transformer 之前，自然语言处理常用 RNN。RNN 的特点是按顺序一个 token 一个 token 地处理文本，后面的计算依赖前面的隐藏状态。这种方式有两个明显问题：

- 难并行：必须按顺序处理，训练效率低。
- 长距离依赖弱：文本很长时，前面的信息容易在传递过程中被淡化。

Transformer 的思路是：一次性看完整个序列，通过自注意力直接计算任意两个 token 之间的关系。这样既适合 GPU 并行训练，也更擅长处理长句子中的远距离关系。

### Transformer 的核心组成

Transformer 可以粗略理解为多层堆叠的语义处理器，每一层都在进一步理解 token 之间的关系。

输入侧通常包含：

- Input Embedding：把 token 转成向量，让计算机可以计算。
- Positional Encoding：给 token 加上位置信息，因为 Transformer 本身是同时看所有 token，不天然知道顺序。

核心层通常包含：

- Multi-Head Attention：从多个角度计算 token 之间的关系。
- Feed Forward Network：对每个位置的向量做进一步非线性变换。
- Add & Norm：通过残差连接和层归一化稳定训练。

这些层会堆叠很多次。层数越多，模型越能组合出复杂语义，但训练和推理成本也越高。

### Encoder、Decoder 和 Encoder-Decoder

Transformer 有三类常见结构：

| 结构 | 典型模型 | 适合任务 |
| --- | --- | --- |
| Encoder-only | BERT | 理解、分类、匹配、抽取、Embedding |
| Decoder-only | GPT | 对话、续写、代码生成、工具调用 |
| Encoder-Decoder | 原始 Transformer、T5、BART | 翻译、摘要、改写、输入到输出的转换 |

最短记法：

> BERT 擅长理解，GPT 擅长生成，Encoder-Decoder 擅长把一种序列转换成另一种序列。

ChatGPT 这类生成式对话模型，核心上更接近 Decoder-only 架构：它根据已有上下文不断预测下一个 token，直到生成完整回答。

### Transformer 的优势和边界

Transformer 的优势可以总结成三点：

- 并行计算能力强：不像 RNN 必须顺序处理，Transformer 可以同时处理整个序列，更适合 GPU 大规模训练。
- 长距离依赖能力强：自注意力可以让任意两个 token 直接建立联系，不需要一层层传递隐藏状态。
- 效果好且扩展性强：GPT、BERT、T5 等现代语言模型都建立在 Transformer 或其变体之上。

它的边界也要记住：标准自注意力的计算量通常会随着序列长度平方级增长，所以上下文越长，显存、延迟和成本压力越大。这也是为什么长上下文、RAG 检索和 Context Engineering 在工程里非常重要。

## 核心概念速记

### Token

Token 是模型处理文本的基本单位，会影响上下文长度、调用成本、响应速度和输出上限。

### Context Window

上下文窗口决定一次请求里模型能看到多少内容。系统提示词、用户问题、历史对话、检索资料、工具返回值都会占用上下文窗口。

### Temperature

Temperature 控制输出随机性。它不是控制模型“聪明程度”，而是控制模型在候选 token 之间选择时的发散程度。

### 模型类型

- Base Model：更像文本续写模型。
- Chat Model：面向对话和指令遵循。
- Reasoning Model：更擅长复杂推理和多步任务。
- Embedding Model：把文本映射成向量，用于语义检索。

## LLM 擅长什么

大模型擅长的是语言、知识模式、语义理解和内容生成。常见能力包括：

- 文本理解：阅读、总结、分析文章，做问答和阅读理解。
- 文本生成：写文章、小说、新闻稿、合同、说明文档。
- 机器翻译：在多语言之间转换文本。
- 信息提取：从法律、医学、财务、业务文档中抽取关键信息。
- 代码辅助：生成代码、解释代码、调试 Bug、优化逻辑。
- 广泛知识问答：回答百科、历史、科技、经济等通用知识问题。
- 多模态能力：识别或生成图像、语音、视频等内容。
- 长文档处理：总结会议纪要、论文、产品文档。
- 结构化转换：把非结构化文本转成表格、JSON 或固定字段。

但“擅长”不等于“可靠”。只要涉及事实准确性、实时数据、私有知识、精确计算、权限判断和业务动作，就必须引入工程手段兜底。

## LLM 的局限性

### 幻觉问题

幻觉是指模型生成了看起来合理、语气肯定，但事实并不可靠的内容。

根本原因是：LLM 的目标是生成语言上最可能、最连贯的 token，而不是内置事实校验机制。它可能从训练数据中学到了相关表达模式，但并不知道当前结论是否真实、是否有依据、是否适用于你的业务场景。

所以，当你让模型写“基于中医理论实现星际穿行的论文”时，它可能会输出一篇结构完整、语言自然的文章，但这不代表内容成立。

面试表达：

> 幻觉的本质不是模型不会说话，而是它太会说话。它会优先生成语言上连贯的答案，但没有天然的事实校验、证据引用和业务规则校验能力。

### 不擅长精确计算

大模型不是计算器。它可以根据训练数据和上下文生成“看起来像计算过程”的文本，但不保证数值计算一定正确。

例如“3.9 和 3.11 哪个大”这类问题，模型可能因为语言模式、数字 token 表示和上下文联想而答错。金额、库存、利息、税费、折扣、账单等场景更不能直接依赖模型口算。

工程上应该让模型负责理解意图和生成解释，让代码、数据库、规则引擎或工具负责真正计算。

### 缺少实时性

模型训练完成后，参数中的知识就固定了。训练截止时间之后发生的新闻、价格、政策、版本变化，模型本身并不知道。

如果模型能回答最新新闻，通常是因为产品接入了联网搜索、浏览器、数据库或其他外部工具，而不是模型参数天然知道最新事实。

### 私有知识缺失

模型训练数据主要来自公开语料，但企业内部制度、产品文档、私有代码、业务 SOP、客户数据等不在公开训练集中。即使模型通用能力很强，也不能凭空知道某个公司的内部规则。

这类问题不是“模型不够聪明”，而是它没有看到你的私有资料。

### 工程不可控

除了上面几个典型问题，LLM 应用还会遇到这些工程风险：

- 输出格式不稳定，程序难解析。
- 上下文过长，成本和延迟升高。
- 用户输入可能包含 Prompt Injection。
- 工具调用可能越权或参数错误。
- 多轮对话可能被历史消息污染。

## 常见解决方案

大模型应用开发的本质，是借助大模型这个“大脑”理解语言和规划任务，再通过工程系统给它补上知识、工具、记忆、权限和控制。

| 问题 | 常见方案 | 说明 |
| --- | --- | --- |
| 幻觉 | Prompt Engineering、RAG、引用校验、评测集 | 提醒模型按资料回答，并提供可追溯证据 |
| 精确计算 | Function Calling、代码执行、数据库、规则引擎 | 模型识别意图，计算交给确定性系统 |
| 实时性 | 搜索工具、API 工具、MCP Server | 通过外部工具获取最新信息 |
| 私有知识 | RAG、企业知识库、向量检索、权限过滤 | 把内部资料检索后注入上下文 |
| 固定风格或格式 | Prompt、结构化输出、微调 | 稳定输出格式、话术、分类方式 |
| 外部动作 | Function Calling、MCP、工作流 | 让模型调用后端服务，但动作由代码执行 |
| 质量回归 | Harness、离线评测、线上监控 | 让 Prompt、RAG、Agent 的变化可度量 |

### Prompt Engineering

Prompt Engineering 负责把任务、角色、上下文、约束和输出格式讲清楚。它可以降低幻觉、提升格式稳定性，但不能凭空补充模型不知道的事实。

### RAG

RAG 负责把外部知识检索出来，再作为上下文交给模型。它适合解决私有知识、知识更新、答案可追溯问题。

### Function Calling

Function Calling 让模型选择要调用的函数并生成参数，真正执行由后端代码完成。它适合解决精确计算、查数据库、调业务接口、执行动作等问题。

### MCP

MCP 可以理解为把工具、资源和提示词标准化暴露给模型应用的一套协议。它适合让多个客户端复用同一批工具能力。

### 微调

微调适合让模型学习稳定格式、固定风格、领域表达或分类模式。但它不适合解决实时数据和频繁变化的企业知识。

## Java 后端视角的理解

可以把大模型应用拆成几个部分：

- 大脑：LLM，负责理解自然语言、生成内容、做初步推理和规划。
- 四肢：Function Calling、MCP、业务 API、数据库、搜索工具，负责执行动作和获取外部信息。
- 记忆和知识库：RAG、向量库、文档库、会话记忆，负责补充模型不知道的信息。
- 控制系统：Java 后端服务、权限校验、参数校验、工作流、日志、评测和监控，负责保证系统可控、可审计、可回滚。

所以，真正的大模型应用开发，不是把用户问题直接丢给模型，而是设计一套可靠的上下文、工具、知识和控制链路。

典型调用链路可以是：

1. 用户输入问题。
2. 后端识别意图和权限。
3. 按需检索知识库或调用工具。
4. 组装 Prompt 和上下文。
5. 调用 LLM 生成答案或工具调用参数。
6. 后端校验、执行、审计。
7. 将结果返回给用户。

## 面试速记

### 大模型是怎么工作的

大模型本质上是基于上下文预测下一个 token 的概率模型。训练时通过海量语料学习语言模式和语义关系，推理时不断生成 token 并追加回上下文。它看起来像理解，是因为 Transformer 的自注意力机制能捕捉 token 之间的关系。

### Transformer 为什么重要

Transformer 通过自注意力机制同时建模所有 token 之间的关系，相比 RNN 更容易并行训练，也更擅长长距离依赖。GPT、BERT、T5 等现代大模型都基于 Transformer 或其变体。

### 大模型有哪些局限性

主要有幻觉、不擅长精确计算、缺少实时性、缺少私有知识、输出不可控等。根本原因是 LLM 是概率生成模型，不是事实数据库、计算器、实时搜索系统或业务执行系统。

### 工程上怎么解决

用 Prompt Engineering 约束任务和格式，用 RAG 补充私有知识和证据，用 Function Calling/MCP 调用外部工具和业务系统，用代码和数据库做精确计算，用评测和监控保证质量。
