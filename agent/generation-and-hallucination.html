# 大模型生成策略与幻觉抑制

RAG 前半段解决“找到资料”，生成阶段解决“怎么根据资料回答”。如果检索到的 chunk 质量不错，但 Prompt 组织得很差，最终答案仍然可能跑偏。

# 生成阶段的核心问题

RAG 生成阶段主要有三个挑战：

  • 模型是否真的依据资料回答。
  • 答案是否能追溯到来源。
  • 输出格式是否稳定，能不能被前端或后端继续处理。

所以生成阶段不是简单把 chunk 拼到问题后面，而是要明确角色、边界、资料格式和兜底策略。

一个典型反例是：检索到了“试用期员工不享受年假”，但 Prompt 只是写“请回答用户问题”。模型可能结合常识回答“员工一般可以按比例享受年假”。这就是资料有了，但生成阶段没约束住。

# 系统指令

系统指令定义模型的角色和边界。

一个基础写法：

你是一个严谨的知识库问答助手。
你只能根据提供的参考资料回答问题。
如果资料不足，请直接说明“当前资料不足以回答”，不要编造。

# 检索上下文

检索上下文就是召回的 chunk。建议给每段资料加编号和来源信息，方便模型引用。

来源：员工手册 v3.2，第 4 章
内容：……

# 用户问题

用户问题可以使用原始问题，也可以使用查询重写后的问题。

实践中建议同时保留：

  • 原始问题：保留用户真实表达。
  • 改写问题：方便检索和生成理解。

# 幻觉是什么

幻觉就是模型输出了没有依据、错误或张冠李戴的内容。

RAG 场景常见三类：

  • 篡改事实：资料说 A，模型回答成 B。
  • 凭空捏造：资料没有，模型自己补充。
  • 张冠李戴：把资料 1 的结论安到资料 2 的来源上。

幻觉不能只靠“换更强模型”解决，Prompt、检索资料和引用机制都要一起管。

例如资料里只说“北京办公室每周三可申请远程办公”，模型回答成“所有办公室每周三都可以远程办公”，这就是把局部规则泛化成全局规则。

# 抑制幻觉的技巧

# 降低随机性

知识库问答一般使用较低的 temperature。创意写作需要发散，RAG 问答需要稳定。

# 明确资料边界

Prompt 里要明确告诉模型：

参考资料之外的信息不能作为回答依据。
1

# 设计兜底回答

不要让模型在资料不足时硬答。

推荐兜底：

当前资料不足以回答这个问题。建议补充：……
1

兜底不是失败，而是 RAG 系统保持可信度的一部分。

比如用户问“明年公司是否会调整年终奖”，知识库里没有相关公告。系统应该回答“当前资料不足以回答”，而不是根据往年制度推测。

# 要求引用来源

要求模型回答时列出依据，可以显著减少自由发挥。

例如：

回答每个结论时，请标注对应资料编号。
1

# 引用对齐

引用对齐是让答案可追溯的关键。

基本做法：

  1. 检索阶段给每个 chunk 分配稳定编号。
  2. Prompt 中要求模型引用编号。
  3. 后端解析模型输出中的引用。
  4. 前端展示来源标题、页码、链接或原文片段。

注意：引用编号不能只靠模型随便生成，最好由系统提供候选编号。

一个比较稳的展示方式是：答案正文里标 [资料 2]，页面右侧或答案下方展示“资料 2：员工手册 v3.2，第 4 章，原文片段……”。这样用户可以直接点回原文验证。

# 输出约束

RAG 系统常见输出形式：

  • 普通文本：适合聊天和知识库。
  • 列表：适合步骤、规则、对比。
  • JSON：适合后端继续处理。

如果要求 JSON，就要明确字段和异常情况。例如：

只返回 JSON：
{
  "answer": "回答内容",
  "sources": ["资料编号"],
  "confidence": "high | medium | low"
}

# 生成阶段检查清单

  • 是否明确要求只基于资料回答。
  • 是否有资料不足时的兜底话术。
  • 是否控制了 temperature。
  • 是否给 chunk 加了编号和来源。
  • 是否要求答案引用资料编号。
  • 是否限制了输出格式和长度。
  • 是否记录了 Prompt、召回资料和最终答案，方便排查。
