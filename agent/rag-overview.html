# RAG 是什么

# 为什么大模型还需要 RAG

大模型虽然强，但它有几个天然限制：

  • 可能幻觉：没有依据也能编出看起来很真的答案。
  • 知识有时效性：训练完成后，新信息不会自动进入模型。
  • 私有数据不可见：公司文档、内部制度、业务数据库不在模型训练数据里。
  • 垂直领域不够深：模型懂很多，但未必懂你的具体业务。
  • 答案不可追溯：纯模型回答很难说明依据来自哪里。

RAG 就是为了解决这些问题。它的核心思想很简单：不要让模型闭卷考试，而是先把相关资料找出来，再让模型开卷回答。

比如公司 2026 年刚更新了报销制度，通用大模型不可能天然知道这份内部制度。RAG 的做法是：先从公司知识库检索“报销制度 2026 版”的相关段落，再让模型根据这些段落回答，而不是让模型凭训练记忆猜。

# RAG 的一句话解释

RAG 是 Retrieval-Augmented Generation，检索增强生成。

流程可以理解为：

  1. 用户提问。
  2. 系统从知识库里检索相关资料。
  3. 把资料和问题一起交给大模型。
  4. 大模型基于资料生成答案。

模型负责“组织语言和推理”，知识库负责“提供依据”。

一个实际例子：

用户问题：出差打车能不能报销？
检索资料：差旅制度中关于市内交通费的条款。
模型回答：可以，但需要满足制度中的时间、地点和票据要求，并引用对应条款。

# RAG 标准流程

# Ingest：导入数据

把 PDF、Word、网页、Markdown、数据库记录等原始资料导入系统。

这一阶段最重要的是拿到干净文本。文档解析质量差，后面的检索和回答都会受影响。

# Chunk：切分文档

长文档不能直接塞给模型，也不适合整篇建索引，所以要切成小块。

每个 chunk 应该尽量表达一个相对完整的语义片段。切得太碎会丢上下文，切得太大又会影响检索精度。

例如一份员工手册有 80 页，用户只问“年假怎么计算”。如果整本手册作为一个 chunk，检索结果太粗；如果把一句话切成一个 chunk，又可能丢掉上下文。比较合适的是按章节、段落切，让“年假规则”相关内容形成几个完整片段。

# Embed：文本向量化

用 Embedding 模型把每个 chunk 转成向量。

向量可以理解成一组数字坐标，语义越接近的文本，向量距离越近。

# Index：存入向量数据库

把 chunk 的文本、向量和元数据一起存入向量数据库。

元数据包括文件名、页码、章节、更新时间、权限标签等。

# Retrieve：检索相关内容

用户提问时，先把问题也转成向量，再到向量库里找相似 chunk。

真实项目里通常不会只靠向量检索，还会结合关键词检索、过滤条件和重排序。

# Answer：生成答案

把检索到的资料拼进 Prompt，让大模型生成最终回答。

这一步要明确要求模型“只根据资料回答”，否则模型可能又开始自由发挥。

生成阶段最好让模型带来源。例如“根据《员工手册》第 3.2 节，正式员工入职满一年后可享受年假”。这样用户不仅看到答案，也知道答案从哪里来。

# RAG 适合什么场景

RAG 特别适合这些场景：

  • 企业内部知识库
  • 智能客服
  • 合同、法律、医疗等专业问答
  • 技术文档助手
  • 代码仓库问答
  • 产品手册查询
  • 规章制度问答

共同特点是：答案必须基于一批可更新、可追溯的外部资料。

# RAG 的优点

  • 知识可更新：更新知识库即可，不一定要重新训练模型。
  • 答案可追溯：可以返回引用来源。
  • 成本较低：相比微调或训练模型，工程成本更可控。
  • 适合私有数据：企业内部文档可以进入自己的知识库。

# RAG 的难点

RAG 看起来流程简单，真正做好并不容易。常见问题包括：

  • 文档解析不干净。
  • Chunk 切分不合理。
  • 元数据缺失，无法过滤和追溯。
  • 向量模型选型不合适。
  • 检索召回不准。
  • Prompt 没有限制住模型。
  • 答案无法评估好坏。

所以 RAG 项目的核心不是“调一个大模型接口”，而是围绕数据、检索、生成做一整套工程。
