# API 调用与 Prompt 工程

# OpenAI 兼容协议

现在很多模型厂商都会兼容 OpenAI 风格的接口协议。好处是：你理解一套请求结构，就能较容易地切换不同模型服务。

一个典型 Chat API 请求包含：

  • model：指定调用哪个模型。
  • messages：对话消息数组。
  • temperature：控制随机性。
  • max_tokens：限制最大输出长度。
  • top_p：另一种控制采样范围的参数。
  • stream：是否启用流式输出。

messages 是最重要的部分，常见角色有：

  • system：告诉模型身份、规则、边界。
  • user：用户输入。
  • assistant：模型历史回复。

一个最小对话可以这样理解：

[
  {"role": "system", "content": "你是一个严谨的 Java 面试题讲解助手。"},
  {"role": "user", "content": "HashMap 为什么线程不安全？"}
]

system 像规则说明，user 才是这次真正的问题。多轮对话时，历史里的 assistant 回复也会继续放进数组里，让模型知道前面聊过什么。

# 非流式调用

非流式调用就是：请求发出去，等模型生成完整答案后一次性返回。

它适合：

  • 后台任务
  • 文档摘要
  • 分类打标
  • 结构化抽取
  • 对实时体验要求不高的接口

优点是实现简单，后端拿到完整结果后再处理即可。缺点是用户要等到全部生成完成，长回答时体验会比较慢。

比如后台每天凌晨批量总结 100 篇工单，就很适合非流式。用户不盯着屏幕等结果，系统只需要最终拿到完整摘要。

# 流式调用

流式调用会把模型生成的内容一段一段返回，页面上看起来像模型正在打字。

它适合：

  • 聊天机器人
  • 智能客服
  • 编程助手
  • 需要即时反馈的交互式产品

常见实现方式是 SSE，也就是 Server-Sent Events。后端持续接收模型返回的增量内容，再转发给前端。

比如用户在聊天框里问“帮我分析这段报错”，模型可能要生成几百字。如果等全部生成完再返回，页面会卡住几秒；流式返回时，用户马上能看到第一句话，体验会自然很多。

工程上要注意：

  • 前端要能逐段追加内容。
  • 后端要处理连接关闭、超时、异常中断。
  • 流式内容可能不是完整 JSON，需要按事件片段解析。
  • 最终仍要记录完整回答，方便日志、审计和问题排查。

# Prompt 为什么重要

Prompt 不是“随便问一句”。它更像给模型的一份任务说明书。说明越清楚，模型越稳定。

一个好的 Prompt 通常包含五个部分：

  • 角色：你是谁。
  • 任务：你要做什么。
  • 约束：不能做什么，优先级是什么。
  • 输入：给模型哪些资料，资料边界在哪里。
  • 输出：希望模型按什么格式回答。

# Prompt 五要素

# 角色

角色用来限制模型的回答视角。

例如：

你是一个企业知识库问答助手，只能根据给定资料回答问题。
1

角色不需要写得很玄，关键是明确边界。

# 任务

任务要用明确的动词描述，例如：

  • 总结以下文档。
  • 提取合同中的风险条款。
  • 根据资料回答用户问题。
  • 判断用户意图属于哪个分类。

少用“帮我看看”“分析一下”这种模糊说法。

# 约束

约束用来减少模型自由发挥。

常见约束包括：

  • 不知道就说不知道。
  • 不要编造资料中不存在的内容。
  • 回答必须引用来源。
  • 输出控制在 200 字内。
  • 只返回 JSON，不要解释。

RAG 场景里最重要的约束是：只根据检索资料回答。

# 输入

输入要有清晰边界。推荐用分隔符把不同内容包起来，避免模型把指令和资料混在一起。

用户问题：
{{question}}

参考资料：
---
{{context}}

例如：

# 常用技巧

# 明确性

用祈使句，不用含糊问题。

不推荐：

你能不能简单讲讲？
1

推荐：

用 3 个要点解释这个概念，每个要点不超过 50 字。
1

# 具体性

告诉模型处理逻辑，而不是只给目标。

例如做知识库问答时，可以要求：

先判断参考资料是否包含答案。包含时回答问题并列出依据；不包含时说明资料不足。
1

# 分步引导

复杂任务可以拆步骤，但不一定要让模型展示推理过程。

推荐写法：

请按步骤完成：先识别问题类型，再查找相关资料，最后给出简洁答案。
1

# 示例驱动

给 2 到 3 个例子，模型会更容易模仿格式和判断标准。分类、抽取、改写任务尤其适合 few-shot。

例如做意图分类时，直接说“判断用户意图”比较空；给两个例子后会稳定很多：

示例 1：
用户：我想查一下订单 10086 到哪了
输出：tool_invocation

示例 2：
用户：公司的年假规则是什么？
输出：knowledge_retrieval

1. 只能根据参考资料回答。
2. 如果资料不足，直接说明“当前资料不足以回答”。
3. 不要编造来源。
4. 回答要简洁，必要时列出依据。
