# RAG 知识库入库

知识库入库是 RAG 的地基。一个常见误区是：以为 RAG 只是把文件读出来，然后丢进向量库。真实项目里，文档解析、分块、元数据管理才是最容易出问题的地方。

## 文档解析为什么麻烦

现实里的文档并不干净：

- PDF 可能是文本 PDF，也可能是扫描图片。
- Word 里可能有表格、页眉页脚、批注、图片。
- 文件后缀可能是错的。
- 中文编码可能乱码。
- 表格抽出来后可能失去结构。
- 图片里的文字需要 OCR。

如果解析阶段就丢内容、乱顺序、乱码，后面向量化和检索都会被污染。

一个常见坑是扫描版 PDF。文件后缀是 `.pdf`，但里面其实是一张张图片。你用普通文本抽取工具可能得到空字符串，后面再怎么调 Embedding 和检索都没用，因为知识库根本没有真正入库。这种情况就要先走 OCR。

## Apache Tika 的价值

Apache Tika 是一个文档解析工具，适合用来做 RAG 入库的第一步。

它主要解决三类问题：

- MIME 类型检测：判断文件真实类型，而不是只看后缀。
- 文本抽取：从 PDF、Word、Excel、HTML 等文件中抽出文本。
- 元数据抽取：拿到作者、创建时间、页数、标题等信息。

Tika 不是万能的。遇到扫描 PDF 或图片文字时，还需要 OCR；遇到复杂表格时，可能还要做专门的结构化解析。

## Chunk 是什么

Chunk 就是文档切分后的文本块。

为什么要分块：

- 大模型上下文有限，不能把所有文档都塞进去。
- 检索需要更细粒度，整篇文档太粗。
- 答案引用需要定位到具体片段。

一个好的 chunk 应该尽量满足：

- 语义完整。
- 长度适中。
- 保留必要上下文。
- 能追溯回原文位置。

举例：原文是“正式员工入职满一年后，每年享有 5 天年假。年假不可跨两个自然年度累计。”这两句话最好在同一个 chunk 里。如果切成两块，用户问“年假能不能跨年累计”时，模型可能只拿到后半句，却不知道这是在说正式员工的年假规则。

## 关键参数

### chunkSize

`chunkSize` 是每个 chunk 的目标大小。

太小的问题：

- 上下文不完整。
- 容易切断一个概念。
- 模型拿到资料后看不懂前因后果。

太大的问题：

- 检索不精准。
- 占用更多上下文。
- 噪声变多。

### overlap

`overlap` 是相邻 chunk 之间的重叠内容。

它的作用是避免重要信息刚好被切断。例如一个定义在上一段，解释在下一段，如果完全不重叠，检索时可能只召回半截。

overlap 也不能太大，否则向量库里会出现大量重复内容。

## 常见分块策略

### 固定大小分块

按固定字符数或 token 数切。

优点是简单稳定，容易实现。缺点是不理解语义，可能把标题、段落、代码块切断。

适合快速原型，不适合作为复杂知识库的最终方案。

### 重叠分块

在固定大小基础上增加 overlap。

优点是减少上下文断裂。缺点是重复内容增多，存储和检索成本也会上升。

### 递归分块

按段落、句子、标点等层级逐步切分。优先保留自然结构，实在太长再继续切。

这是通用 RAG 项目里比较推荐的策略。

### 语义分块

根据语义相似度或主题变化切分。

优点是更贴近内容含义。缺点是实现复杂，成本更高，也更依赖模型效果。

### 混合分块

先按标题、段落等结构切，再用大小限制兜底。实际项目里常用这种方式。

例如 Markdown 文档可以先按 `#`、`##` 标题切；如果某个标题下内容太长，再按段落或 token 数继续拆。这样既保留章节语义，又不会让单个 chunk 太大。

## 元数据为什么重要

只存 chunk 文本是不够的。元数据相当于给每个 chunk 贴标签。

没有元数据会带来几个问题：

- 用户问“依据是什么”，系统答不上来。
- 权限无法控制，不同部门可能看到不该看的内容。
- 答案出错后，找不到是哪份文档、哪个片段导致的。
- 文档更新后，旧版本无法清理。

## 常见元数据字段

### 文档标识

常见字段：

- `doc_id`
- `file_name`
- `source_url`
- `source_type`

作用是知道 chunk 来自哪里。

### 结构信息

常见字段：

- `title`
- `heading_path`
- `page_number`
- `section_no`
- `chunk_index`

作用是让答案能追溯到章节、页码和位置。

### 时间版本

常见字段：

- `created_at`
- `updated_at`
- `effective_at`
- `expired_at`
- `version`

制度、合同、产品文档经常有版本问题。检索时要避免召回已经失效的资料。

### 权限控制

常见字段：

- `department`
- `role`
- `acl`
- `visibility`

企业知识库一定要考虑权限过滤。不要等到检索后再用 Prompt 让模型“不要回答”，正确做法是在检索阶段就过滤掉用户无权访问的 chunk。

比如财务制度里有“全员可见”的报销规则，也有“财务部内部可见”的审核口径。普通员工提问时，检索 SQL 或向量过滤条件就应该排除财务内部 chunk，而不是把资料交给模型后再提醒它别说。

### 位置追溯

常见字段：

- `start_offset`
- `end_offset`
- `page_number`
- `paragraph_index`

作用是支持引用、定位和排错。

## 入库流程建议

一个比较稳的入库流程：

1. 检测文件类型。
2. 抽取文本和基础元数据。
3. 清洗无效内容，比如页眉页脚、重复水印。
4. 按结构进行分块。
5. 给 chunk 补充文档来源、章节、页码、权限、版本等元数据。
6. 调用 Embedding 模型生成向量。
7. 写入向量数据库。
8. 记录入库日志，方便后续重建和排查。
