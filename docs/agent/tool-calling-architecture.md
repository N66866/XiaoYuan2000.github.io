# 工具调用架构与安全保障

Function Calling 解决“模型怎么表达要调用工具”，但真正做成生产系统，还需要一套工具调用架构。工具定义、参数、返回值、错误处理、安全、监控，都会直接影响 Agent 是否可靠。

## 工具定义原则

### 单一职责

一个工具只做一件事。

不推荐：

```text
manage_order：查询订单、取消订单、申请退款都放在一起
```

推荐：

```text
query_order_status：查询订单状态
cancel_order：取消订单
create_refund_request：创建退款申请
```

工具越清晰，模型越不容易误调用。

### 参数最小化

只让模型填写真正必要的参数。

例如查询订单状态，通常只需要 `orderId`。不要把 `userName`、`phone`、`address` 都设计成参数，否则模型更容易填错，也增加敏感信息风险。

### 幂等性

能设计成幂等的工具尽量幂等。

查询类工具天然幂等；写操作要谨慎，比如创建工单、取消订单、发起退款。对写操作可以增加 `requestId`，防止模型或网络重试导致重复执行。

### 返回值结构化

工具返回值不要只返回一段自然语言，最好返回结构化对象。

```json
{
  "success": true,
  "data": {
    "orderId": "10086",
    "status": "DELIVERING",
    "estimatedArrival": "2026-05-14"
  }
}
```

结构化结果更适合交给模型做第二轮回答，也方便后端记录和测试。

## 工具描述怎么写

工具描述要包含三件事：

- 工具做什么。
- 什么时候该用。
- 什么时候不该用。

不推荐：

```text
查询订单。
```

推荐：

```text
根据订单号查询订单的支付状态、发货状态和预计送达时间。当用户询问某个具体订单当前状态时使用。不能用于查询退款规则。
```

多工具场景里，描述还要拉开差异。比如“查询退款规则”属于 RAG，“创建退款申请”属于工具调用，两者边界要写清楚。

## 参数设计

### 必填与可选

必填参数必须明确，缺失时应该追问，不要让模型猜。

例如取消订单需要 `orderId`。如果用户只说“帮我取消订单”，系统应该追问“请提供订单号”。

### 默认值

默认值适合低风险参数。

例如查询订单列表时，`pageSize` 可以默认 10；但“退款金额”绝不能默认。

### 参数校验

工具执行前必须校验参数：

- 类型是否正确。
- 格式是否正确。
- 是否越权。
- 是否包含注入风险。
- 是否超出业务范围。

模型输出的 JSON 只是建议，不能当作可信输入。

### 命名规范

参数名要稳定、直观，尽量和业务系统一致。

比如统一使用 `orderId`，不要一会儿叫 `order_id`，一会儿叫 `id`，一会儿叫 `orderNo`。

## 返回值规范

建议统一返回结构：

```json
{
  "success": true,
  "code": "OK",
  "message": "success",
  "data": {}
}
```

失败时：

```json
{
  "success": false,
  "code": "ORDER_NOT_FOUND",
  "message": "订单不存在",
  "data": null
}
```

返回值的信息密度要适中。不要把整个数据库对象原样返回给模型，尤其是手机号、身份证、地址、内部备注这类敏感字段。

## 错误处理

### 超时

每个工具都要有超时时间。不要让一个慢接口拖住整个 Agent 回复。

比如查询订单 2 秒内没返回，可以提示“订单系统暂时响应较慢”，而不是让用户一直等。

### 重试

只对安全、幂等的操作重试。

查询订单失败可以重试；创建退款申请失败不能随便重试，否则可能重复创建。

### 降级

工具不可用时，可以降级：

- 查实时订单失败：提示稍后再试。
- 查规则失败：尝试从知识库回答。
- 外部系统不可用：转人工或创建待处理任务。

### 熔断

如果某个工具持续失败，就暂时停止调用它，避免把故障扩大。

## 安全保障

工具调用的安全边界一定在后端，不在模型。

必须做：

- 权限控制：用户有没有资格调用这个工具。
- 参数校验：防止越权、注入和非法输入。
- 敏感信息脱敏：不要把不必要的隐私交给模型。
- 审计日志：记录谁在什么时间调用了什么工具。
- 高风险操作确认：删除、退款、取消订单等操作要二次确认。

例如用户说“把这个客户账号删掉”，模型可以识别为删除账号意图，但系统必须检查权限，并要求用户确认。

## 监控与告警

工具调用至少要监控四类指标：

- 调用量：工具被调用多少次。
- 成功率：成功和失败比例。
- 延迟：平均耗时、P95、P99。
- 错误分布：哪些错误码最多。

同时要有链路追踪：从用户问题、模型 tool_call、工具执行、最终回答串起来。否则线上出问题时，很难判断是模型选错工具，还是工具本身失败。

## 测试策略

### 单元测试

验证工具函数本身逻辑是否正确。

### 集成测试

验证模型是否能正确选择工具、生成参数、处理工具结果。

### 压力测试

验证高并发下工具接口、线程池、连接池是否稳定。

### A/B 测试

比较不同工具描述、参数设计、路由策略对调用准确率的影响。

## 一个优化例子

初始工具：

```text
name: order
description: 处理订单
params: { "id": "string" }
```

问题很多：名字太泛、描述不清、参数含义不明确。

优化后：

```text
name: query_order_status
description: 根据订单号查询订单的支付状态、发货状态和预计送达时间。当用户询问具体订单当前状态时使用，不能用于取消订单或申请退款。
params: { "orderId": "string" }
```

模型会更容易判断什么时候调用，也更容易生成正确参数。

## 工具调用检查清单

- 工具是否单一职责。
- 描述是否写清使用边界。
- 参数是否足够少、足够明确。
- 写操作是否幂等或有二次确认。
- 返回值是否结构化。
- 错误码是否统一。
- 是否有超时、重试、降级、熔断。
- 是否做了权限、脱敏、审计。
- 是否有调用量、成功率、延迟、错误监控。

工具调用不是“让模型调接口”这么简单。真正可靠的 Agent，需要把工具当成一套后端能力来设计和治理。
