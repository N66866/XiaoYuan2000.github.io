# Function Calling 与工具调用

RAG 能让模型查知识库，但它本质上还是“回答问题”。如果用户想让系统去做一件事，比如查订单、创建工单、发通知、调用业务接口，单靠 RAG 就不够了。

Function Calling 解决的就是这个问题：让模型不仅能回答，还能判断什么时候该调用工具。

## RAG 的能力边界

RAG 擅长：

- 查制度。
- 查文档。
- 总结资料。
- 根据知识库回答问题。

但它不擅长：

- 查询实时订单状态。
- 修改数据库。
- 创建业务流程。
- 调用外部系统。
- 根据用户输入执行动作。

这些动作需要真实接口参与，不能让模型凭空回答。

## Function Calling 是什么

Function Calling 的本质不是模型真的执行函数，而是模型输出“我想调用哪个函数，以及参数是什么”。

真正执行函数的是你的程序。

完整流程通常是：

1. 后端把可用工具列表发给模型。
2. 用户提出问题。
3. 模型判断是否需要调用工具。
4. 如果需要，模型返回 `tool_calls`。
5. 后端解析工具名和参数，执行真实函数。
6. 后端把函数执行结果再发给模型。
7. 模型基于执行结果生成最终回复。

模型负责决策和组织语言，系统负责执行和安全控制。

以“查订单”为例：

```text
用户：帮我查一下订单 10086 到哪了
模型：需要调用 query_order_status，参数 orderId=10086
后端：调用订单系统，得到“运输中，预计明天送达”
模型：组织成自然语言回复用户
```

## 工具定义

工具定义一般包含：

- 工具名。
- 工具描述。
- 参数 schema。

其中描述非常重要。模型会根据描述判断什么时候使用这个工具。

一个工具描述应该说明：

- 这个工具做什么。
- 什么时候该用。
- 什么时候不该用。
- 参数分别代表什么。

不要写太短，比如“查询订单”。更好的写法是：

```text
根据订单号查询订单的当前状态、支付状态和物流状态。当用户询问某个具体订单的状态时使用。
```

## 参数 Schema

Function Calling 通常用 JSON Schema 描述参数。

它的价值是让模型输出结构化参数，而不是一段自然语言。

例如订单查询工具可能需要：

```json
{
  "orderId": "订单号"
}
```

参数越清晰，后端解析越稳定。

如果用户只说“帮我查下订单”，但没有订单号，模型不应该编一个 `orderId`。更好的行为是返回澄清问题：“请提供订单号”。这类参数缺失要在工具调用策略里明确处理。

## 第二轮请求

工具执行完成后，不要直接把原始接口结果丢给用户。通常要把结果再交给模型，让模型转换成自然语言。

例如：

```text
工具返回：{"status":"PAID","shippingStatus":"DELIVERING"}
最终回答：订单已支付，目前正在配送中。
```

这就是第二轮请求的意义：模型把工具结果整理成用户能理解的话。

## Function Calling 与 RAG 的关系

RAG 和 Function Calling 是互补的。

常见组合：

- 用户问制度：走 RAG。
- 用户查订单：走工具调用。
- 用户问“我的订单为什么不能退款”：先查订单状态，再查退款规则，最后综合回答。

第三种就是 Agent 系统里很常见的混合场景：工具提供实时状态，RAG 提供规则依据。

例如用户问“我的订单 10086 已经发货了，还能不能退款？”系统可以先调用订单工具查到“已发货”，再从知识库检索“已发货订单退款规则”，最后综合回答“可以申请退货退款，但需要收到货后按退货流程处理”。

## 工具调用策略

工具调用不能完全放飞模型。建议加一层路由和策略控制。

可以按优先级判断：

1. 用户是否明确要求实时业务数据。
2. 是否能从问题中提取工具必需参数。
3. 用户是否有权限调用该工具。
4. 工具失败时是否可以降级到知识库回答或澄清。

如果参数缺失，不要让模型乱猜，应该追问。

## 安全边界

Function Calling 最大的风险是：模型输出了工具调用意图，但这个意图不一定安全。

后端必须负责：

- 参数校验。
- 权限校验。
- 工具白名单。
- 调用频率限制。
- 敏感操作二次确认。
- 日志审计。

模型不能直接拥有执行权限。模型只提出调用建议，系统决定能不能执行。

尤其是写操作，比如“帮我取消订单”“帮我删除账号”“给客户退款 500 元”。这类工具调用通常需要用户确认、权限校验和审计日志，不能因为模型判断要调用就直接执行。

## 局限性

Function Calling 工具一多，会出现几个痛点：

- 工具定义维护成本高。
- 工具描述不清会导致误调用。
- 多语言、多系统集成不统一。
- 权限和审计容易散落在各处。
- 工具变更后客户端同步麻烦。

这些问题就是 MCP 要进一步解决的方向。

## 实践建议

第一版可以先这样做：

- 只开放少量高价值工具。
- 工具描述写清楚使用边界。
- 所有工具调用经过统一 Router。
- 工具执行前做参数和权限校验。
- 工具执行结果结构化返回。
- 记录完整链路：用户问题、模型 tool_call、工具结果、最终回答。

Function Calling 是从 RAG 走向 Agent 的关键一步：系统开始不只是“知道”，而是能“行动”。
