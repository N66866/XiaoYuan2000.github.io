# Java Agent：知识库文档上传与分块接口

知识库文档处理建议拆成两个接口：上传接口和开始分块接口。上传解决“文件进入系统”，分块解决“文件进入 RAG 检索体系”。这两个动作职责不同、耗时不同、失败处理也不同。

# 文档上传接口

上传接口的目标是创建一条文档记录，并把原始文件放到对象存储。

输入通常包括：

  • knowledgeBaseId
  • sourceType
  • file
  • sourceUrl
  • processMode
  • chunkConfig
  • scheduleConfig
  • documentName

其中 sourceType 可以是：

  • FILE：本地上传。
  • URL：远程文件。

processMode 可以是：

  • CHUNK：普通分块。
  • PIPELINE：进入复杂处理流水线。

# upload 核心流程

Controller 接收请求
  -> 解析 sourceType
  -> 校验参数
  -> 解析定时同步配置
  -> 解析处理模式配置
  -> 上传或抓取文件
  -> 构建文档记录
  -> 插入数据库
  -> 返回 documentId

# FILE 类型处理

本地文件上传流程：

校验 MultipartFile
  -> 校验大小、扩展名、文件名
  -> 上传对象存储
  -> 计算文件 hash
  -> 保存 objectKey

注意点：

  • 文件名不能直接信任用户输入。
  • 文件类型不要只看后缀。
  • 大文件上传要避免一次性读入内存。
  • 对象存储成功但 DB 失败时，要有清理或补偿。

# URL 类型处理

URL 文件不是直接保存 URL 就完事。最好先抓取一份到对象存储。

流程：

HEAD 探测
  -> 校验 Content-Type / Content-Length
  -> 流式下载到临时文件
  -> 上传对象存储
  -> 计算 hash
  -> 保存 sourceUrl 和 objectKey

为什么要流式下载到临时文件？

  • 避免远程流读取失败后对象存储里出现半截文件。
  • 方便计算 hash。
  • 方便后续重试和清理。

# upload 为什么不加大事务

上传涉及对象存储和数据库。对象存储不参与 MySQL 事务。

如果把整个 upload 方法都包在 @Transactional 里，不能解决对象存储一致性，反而可能让数据库事务持有太久。

更合理的做法：

  • 对象存储上传先完成。
  • DB 插入用短事务。
  • DB 插入失败时清理对象存储。
  • 清理失败时记录补偿任务。

# 为什么上传和分块拆开

拆成两个接口的好处：

  • 上传响应更快。
  • 分块可以异步执行。
  • 用户可以先检查文档信息。
  • 分块失败不影响原始文件保存。
  • 后续可以支持重新分块。
  • 分块配置可以在上传后调整。

如果 upload 里自动触发分块，接口会变得很重，失败边界也会混在一起。

# 开始分块接口

开始分块接口的目标是把文档从“已上传”推进到“异步处理队列”。

核心流程：

校验文档归属和状态
  -> CAS 更新状态为 CHUNKING
  -> 注册或更新定时任务
  -> 发送事务消息
  -> 返回任务已提交

# CAS 状态更新

多个用户或任务可能同时触发分块。可以用 CAS 防并发：

UPDATE t_knowledge_document
SET status = 'CHUNKING'
WHERE id = ?
  AND status IN ('UPLOADED', 'FAILED', 'CHUNKED')

2
3
4

如果影响行数为 0，说明状态已经被别人改了，就不要重复提交任务。

CAS 比分布式锁更轻，适合这种“抢一次状态”的场景。

# 为什么用事务消息

开始分块涉及两个动作：

  • 更新数据库状态。
  • 发送 MQ 消息。

如果 DB 成功但 MQ 失败，文档会卡在 CHUNKING。如果 MQ 成功但 DB 回滚，Consumer 会处理一个不存在或状态不对的任务。

事务消息或可靠消息可以降低这种不一致风险。

# Consumer 异步处理链路

Consumer 收到消息后执行：

创建分块日志

  -> 文档解析
  -> 文本清洗
  -> 分块
  -> Embedding
  -> 写入 chunk 表
  -> 写入向量库
  -> 更新文档状态
  -> 更新分块日志

# 原子性写入设计

分块结果要写两个地方：

  • MySQL chunk 表。
  • 向量数据库。

向量库通常不参与 MySQL 事务。可以采用“先删后插 + 补偿”的策略。

常见流程：

开启 DB 事务
  -> 删除旧 chunk
  -> 插入新 chunk
  -> 提交 DB
写向量库
  -> 删除旧向量
  -> 插入新向量
失败则记录补偿任务

重点不是追求绝对分布式事务，而是让失败可检测、可重试、可修复。

# 为什么先删后插

重新分块时，旧 chunk 已经不可靠。

先删后插的好处：

  • 避免旧 chunk 和新 chunk 混在一起。
  • chunkIndex 重新计算更简单。
  • 向量库可以按 documentId 清理旧索引。

风险是中途失败可能短时间无可用 chunk，所以需要状态字段和日志保护。

# 分块日志

分块日志可以记录：

  • document_id
  • task_id
  • status
  • phase
  • chunk_count
  • error_message
  • started_at
  • finished_at

它的价值：

  • 用户能看到处理进度。
  • 开发能定位失败阶段。
  • 历史记录可追溯。
  • 后续可以做重试和统计。

# 接口设计 FAQ

# 为什么不在 upload 中自动分块

因为上传和分块失败边界不同。拆开后系统更容易做重试、状态展示和人工干预。

# 为什么 startChunk 不加大事务

startChunk 主要做状态抢占和消息发送。大事务无法覆盖 MQ，应该用事务消息或可靠消息机制。

# 如果 Consumer 失败怎么办

更新分块日志和文档状态为 FAILED，保留错误原因，允许用户重新触发。

# 为什么 chunk 表和向量库要一起考虑

用户检索时依赖向量库，管理端展示依赖 MySQL。两边不一致会导致“能搜到但后台看不到”或“后台有但搜不到”。
